# LINE-like 即時語音 Agent 技術選型表 v1.0

> 專案：江彬語氣靈 (JianBin Voice Spirit)
> 目標：無限接近 LINE 語音通話的體感
> 日期：2026-03-23

---

## 架構總覽

```
[Mobile App / Web App]
  ├─ Mic Capture + AEC / NS / AGC
  ├─ Local playback buffer
  ├─ Interrupt detector
  └─ WebRTC client (LiveKit SDK)
          ↓ WebRTC
[LiveKit Server + coturn]
  ├─ Room / Session / Media routing
  └─ NAT traversal (TURN/STUN)
          ↓
[LiveKit Agents / Orchestrator]
  ├─ Silero VAD
  ├─ Semantic Turn Detection
  ├─ Interrupt Manager
  ├─ Context Builder (← 保留現有業務邏輯)
  └─ Playback Sync
          ↓
[Inference Layer]
  ├─ faster-whisper (STT)
  ├─ Gemini 2.5-Flash (LLM) ← 保留，streaming 化
  └─ CosyVoice (streaming TTS)
          ↓
[Memory / Tooling] ← 全部保留
  ├─ PostgreSQL + pgvector
  ├─ 用戶記憶 / 事件 / 承諾 / 筆記
  └─ Langfuse + OpenTelemetry (新增)
```

## 一、必選 (Must Have)

### 1.1 媒體與連線底座

| # | 元件 | 用途 | 取代現有 | 部署方式 | GPU |
|---|------|------|----------|----------|-----|
| 1 | **LiveKit Server** | 即時音訊房間、媒體路由、session 管理 | ElevenLabs ConvAI WebSocket | Docker self-host 或 LiveKit Cloud | 否 |
| 2 | **coturn** | TURN/STUN，手機在複雜網路下穩定通話 | 無（之前靠 ElevenLabs 處理） | Docker self-host | 否 |
| 3 | **LiveKit Client SDK** | 前端 WebRTC 雙向低延遲音訊 | ElevenLabs JS SDK + MediaRecorder | npm: `livekit-client` / `@livekit/components-react` | 否 |

### 1.2 對話主控骨架

| # | 元件 | 用途 | 取代現有 | 部署方式 | GPU |
|---|------|------|----------|----------|-----|
| 4 | **LiveKit Agents (Python)** | Agent 生命周期、語音互動、turn 邏輯 | turn.py 的通話部分 | 與後端同 Docker 或獨立服務 | 否 |

### 1.3 Turn / 插話 / 通話感核心

| # | 元件 | 用途 | 取代現有 | 備註 |
|---|------|------|----------|------|
| 5 | **Silero VAD** | 第一層快速人聲偵測（~100ms 判定） | ElevenLabs 內建 VAD | LiveKit Agents 內建支援 |
| 6 | **Semantic Turn Detection** | 判斷「停一下」vs「真的講完」 | 無（之前靠靜音判斷） | LiveKit Agents 內建支援 |
| 7 | **Interrupt Manager** | 用戶插話時 AI 立刻停播 + 保留上下文 | 無（之前無此能力） | 自行實作，LiveKit 提供底層 API |

### 1.4 STT（語音轉文字）

| # | 元件 | 用途 | 取代現有 | 部署方式 | GPU |
|---|------|------|----------|----------|-----|
| 8 | **faster-whisper** | 串流 STT，邊聽邊轉文字 | Gemini Flash Lite（批次上傳） | Docker self-host | **是** |

> **為什麼換掉 Gemini STT？**
> 現有方式是錄完整段音檔 → 上傳 → 等結果，延遲 1-3 秒。
> faster-whisper 可用 CTranslate2 加速，串流模式下首字延遲 ~300ms。

### 1.5 LLM（大腦）

| # | 元件 | 用途 | 取代現有 | 部署方式 | GPU |
|---|------|------|----------|----------|-----|
| 9 | **Gemini 2.5-Flash (streaming)** | 串流生成回覆（邊想邊吐 token） | Gemini 2.5-Flash（一次性回覆） | API 調用（保留） | 否 |

> **為什麼保留 Gemini 而非換 vLLM？**
> - 現有 system prompt、context 組裝、人設邏輯已調校成熟
> - Gemini API 的 streaming 模式足夠低延遲（首 token ~200ms）
> - 自架 vLLM 需要高階 GPU (A100/H100)，成本與維運門檻高
> - **建議 v1 先用 Gemini streaming，v2 再評估自架 LLM**

### 1.6 TTS（文字轉語音）

| # | 元件 | 用途 | 取代現有 | 部署方式 | GPU |
|---|------|------|----------|----------|-----|
| 10 | **CosyVoice** | bi-streaming TTS（text-in → audio-out），首包 ~150ms | ElevenLabs TTS API | Docker self-host | **是** |

> **為什麼換掉 ElevenLabs？**
> - ElevenLabs 是雲端 API，延遲受網路影響（~500ms+）
> - CosyVoice 自託管可控制延遲，bi-streaming 首包 ~150ms
> - 中文自然度更高（阿里巴巴針對中文優化）
> - 長期成本更低（無 per-character 計費）

### 1.7 資料庫（保留）

| # | 元件 | 用途 | 現狀 | 改動 |
|---|------|------|------|------|
| 11 | **PostgreSQL + pgvector** | 對話、記憶、用戶資料、向量搜尋 | 已在用 | **不改** |

### 1.8 前端 App

| # | 元件 | 用途 | 取代現有 | 備註 |
|---|------|------|----------|------|
| 12 | **React + LiveKit Components** | 通話 UI（大頭照、狀態、麥克風/掛斷） | Call.tsx + useElevenLabsConvAI | 重寫通話頁，保留登入/文字聊天頁 |
| 13 | **WebRTC AEC/NS/AGC** | 回聲消除、降噪、增益控制 | 無 | 瀏覽器 WebRTC 內建 + LiveKit 增強 |

## 三、先不要碰

| # | 元件 | 理由 |
|---|------|------|
| 19 | **PBX / Twilio / Vonage** | 電話交換機思維，與 LINE-like 目標相悖 |
| 20 | **WebSocket 自建音訊傳輸** | WebRTC 已處理低延遲雙向媒體，不需自建 |
| 21 | **OpenAI Realtime API** | 封閉生態、中文支援不如 Gemini、成本高 |
| 22 | **多語言 TTS (Azure/AWS)** | 先把中文做好，不要分散精力 |
| 23 | **Kubernetes / K8s** | v1 用 Docker Compose 就夠，不要過度工程 |
| 24 | **React Native / Flutter 重寫** | 現有 Capacitor + PWA 夠用，App 殼不是瓶頸 |

│  └──────────┘  └──────────────────────┘ │
│                                         │
│  ┌──────────┐  ┌──────────────────────┐ │
│  │ coturn    │  │ LiveKit Agent        │ │
│  │ :3478     │  │ (VAD+STT+LLM+TTS)   │ │

### 硬體需求

| 服務 | CPU | RAM | GPU | 備註 |
|------|-----|-----|-----|------|
| LiveKit Server | 2 核 | 2GB | 否 | 輕量 |
| coturn | 1 核 | 512MB | 否 | 輕量 |
| FastAPI 後端 | 2 核 | 2GB | 否 | 現有規格 |
| LiveKit Agent | 2 核 | 4GB | 否 | 協調器，不跑推理 |
| faster-whisper | 4 核 | 8GB | **NVIDIA T4+** | large-v3 模型 |
| CosyVoice | 4 核 | 8GB | **NVIDIA T4+** | streaming 模式 |
| PostgreSQL | 2 核 | 4GB | 否 | 現有 |
| **合計** | **17 核** | **~29GB** | **2x T4 (或 1x A10)** | |

### 雲端成本預估 (GCP)

| 方案 | 機器 | 月費 (USD) | 備註 |
|------|------|-----------|------|
| **最省** | 1x g2-standard-8 (L4 GPU) | ~$500-700 | STT+TTS 共用一張 GPU |
| **建議** | 1x n2-standard-4 (CPU) + 1x g2-standard-4 (L4) | ~$400-600 | CPU 跑後端/LiveKit，GPU 跑推理 |
| **對比現有** | Cloud Run + ElevenLabs API | ~$50 基礎 + 用量計費 | 用量大時 ElevenLabs 很貴 |

## 六、風險與緩解

| 風險 | 影響 | 緩解方案 |
|------|------|----------|
| GPU 成本比預期高 | 月費超預算 | v1 先用 Gemini API 做 STT (保留現有)，只換 TTS |
| CosyVoice 中文音色不滿意 | 不像真人 | 備選 Fish-Speech 或保留 ElevenLabs 做 fallback |
| LiveKit 部署複雜 | 開發週期長 | 可先用 LiveKit Cloud (託管版) 快速驗證 |
| faster-whisper 中文準確率 | 辨識錯誤多 | large-v3 模型中文已很好，或改用 SenseVoice |
| 手機端 WebRTC 相容性 | iOS Safari 問題 | LiveKit SDK 已處理主流瀏覽器相容 |

## 七、建議實施順序

```
Phase 0 - 驗證 (1-2 週)
  └─ 用 LiveKit Cloud + 現有 Gemini → 驗證通話體感

Phase 1 - 核心替換 (3-4 週)
  ├─ LiveKit Agent 骨架
  ├─ Silero VAD + Turn Detection
  ├─ LLM streaming 化
  ├─ CosyVoice 串流 TTS
  └─ 前端通話頁重寫

Phase 2 - 自架 STT + 優化 (2-3 週)
  ├─ faster-whisper 部署
  ├─ Interrupt Manager 調校
  ├─ 前端 AEC/NS 優化
  └─ 端到端延遲 < 1 秒

Phase 3 - 觀測 + 調優 (持續)
  ├─ Langfuse 接入
  ├─ 延遲指標儀表板
  └─ Turn detection 微調
```
